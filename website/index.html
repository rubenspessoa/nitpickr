Why self-host

Keep code review private, legible, and aligned with your stack.

nitpickr is for people who want the convenience of AI review without giving up on infrastructure ownership, source visibility, or model choice.

Lowest ongoing cost

Local Docker + tunnel

Run the full stack on your machine, expose a temporary public webhook URL, and mostly pay only for model usage.

  • Best for evaluation and solo use
  • Fastest way to prove value on a real repository
  • No always-on infrastructure bill
Most control

Private VPS

Keep the same API, worker, and Postgres split on infrastructure you own. This is the path when “self-hosted” needs to mean private networking, your own secrets handling, and your own runtime boundaries.

  • Fits teams with existing ops workflows
  • Lets you keep traffic, logs, and secrets on your own box
  • No dedicated nitpickr VPS guide yet, but the runtime shape is already straightforward
Easiest always-on

Railway

Use Railway when you want a stable public webhook endpoint without building your own hosting around the service.

  • Documented end to end in the repository
  • Simple split between public API and private worker
  • Good default for early hosted testing

Private by topology

The worker does the review work in the background, while the API stays a thin webhook ingress. That split makes private hosting on your own VPS practical instead of aspirational.

Readable, source-available codebase

You can inspect the source, audit the review surface, and keep the system legible. The repo is source-available under Elastic License 2.0, with branding governed separately.

No forced managed-service abstraction

nitpickr is not trying to trap teams inside a hosted product. If you start local and later move to Railway or a private VPS, the runtime shape stays the same.

Model control

Choose the model that fits your budget and risk tolerance.

nitpickr is deliberately transparent about the model surface. The documented default is gpt-5-mini, but the review engine is driven by your runtime configuration rather than a locked hosted default.

  • OPENAI_MODEL defines the current review model.
  • OPENAI_BASE_URL lets you point at a compatible endpoint.
  • Prompt optimization modes help keep larger PRs economically sane.
  • More model and provider options are on the roadmap, not hidden behind pricing tiers.
runtime config 
DATABASE_URL=postgres://...

GITHUB_APP_ID=123456
GITHUB_BOT_LOGINS=nitpickr

Practical default: start with gpt-5-mini, then move up only when your review depth justifies the spend.

FAQ

Direct answers for the questions that matter before installation.

Is nitpickr open source?

The project should be described as source-available, not OSI-approved open source. The code is published under Elastic License 2.0, which allows personal and internal commercial use but restricts offering nitpickr itself as a hosted or managed service.

Why would I host this on my own VPS instead of using a hosted tool?

Because some teams want the runtime, logs, secrets, and model traffic under their own operational boundary. nitpickr keeps the deployment shape simple enough that a private VPS remains a real option instead of a marketing checkbox.

What can I control today?

The deployment location, the GitHub App installation scope, the repository instruction surface, the review strictness, and the model endpoint through OPENAI_MODEL and OPENAI_BASE_URL.

What does the product support right now?

Today nitpickr is GitHub App only. It reviews pull requests, publishes inline comments, writes a summary body, and updates a nitpickr / review status check.

How should I evaluate it first?

Start locally with Docker Compose and a public HTTPS tunnel. That gives you the cheapest realistic feedback loop before deciding whether you want an always-on deployment on Railway or a private VPS.
